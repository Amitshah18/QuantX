SECTION 1: STRIPE ANALYSIS (Key Patterns to Adopt)
================================================================================

Based on Stripe's approach to fintech trust-building:

1. CLARITY THROUGH SIMPLICITY
   - Single Call-to-Action per page (not multiple competing buttons)
   - Clear hierarchy: Heading → Description → Action
   - Minimal form fields (only required inputs visible)
   - Example: Stripe's "Connect Wallet" button is THE focal point

2. STATUS TRANSPARENCY
   - Real-time status indicators (Active, Paused, Failed, Pending)
   - Progress indicators showing "which step you're on"
   - Transaction hash displays (proves on-chain execution)
   - Payment history with timestamps (builds confidence)

3. TRUST SIGNALS
   - Wallet address truncation (GABC...XYZ) in top-right
   - Network indicator ("Stellar Testnet")
   - Success states with checkmarks + transaction hash
   - Error states with specific error messages (not generic "Error")

4. CONSISTENT LAYOUT PATTERN
   - Top navbar: Logo | Nav Links | Wallet Status
   - Content area: Card-based layout (max-width: 600px)
   - Footer: Links + Network info
   - Spacing: 24px gaps (Stripe standard)

5. FORM DESIGN PATTERN (Stripe Billing)
   - Labels above inputs (not placeholders)
   - Single column for mobile, auto-layout for desktop
   - Visual feedback on input focus (border color change)
   - Submit button disabled until valid (no orphaned errors)
   - Success state immediately after submit (not delayed)

SECTION 4: FUNCTIONALITY CHECKLIST (Phase 1 Must-Have)
================================================================================

FUNCTION 1: CREATE PLAN
  Frontend Trigger: Form submit on /create
  Form Fields Collected:
    - recipient: string (Stellar address)
    - amount: number (USDC)
    - interval: number (seconds: 60, 604800, 2628000)
    - token: string (contract address)
  Backend/Smart Contract Call:
    POST /api/create-plan OR direct contract call
    Payload: { payer, recipient, amount, token, interval }
  Success Response:
    { transactionHash, status: 'success' }
  UI Flow:
    1. User fills form
    2. Click "Start Subscription"
    3. Show LoadingSpinner ("Processing...")
    4. Call backend
    5. On success: Show TransactionHashDisplay + "Go to Dashboard" button
    6. On error: Show error message in red, allow retry
  Validation:
    - recipient length = 56 and starts with 'G'
    - amount > 0
    - wallet connected (no wallet = disable submit button)

FUNCTION 2: FETCH PLANS
  Frontend Trigger: Load /dashboard (useEffect)
  Backend Call:
    GET /api/plans?userAddress={walletAddress}
  Response Shape:
    [
      {
        id: string (contract agreement ID),
        recipient: string (address),
        amount: number,
        token: string,
        interval: number,
        nextPayment: ISO datetime string,
        status: 'active' | 'paused' | 'cancelled',
        totalPaid: number (optional, for stats),
        createdAt: ISO datetime string
      }
    ]
  UI Flow:
    1. Dashboard loads
    2. Show LoadingSpinner
    3. Fetch plans
    4. Render PlansTable OR empty state ("No plans yet")
    5. Calculate stats (total paid, next payment, count)
    6. Render StatsSection above table

FUNCTION 3: EXECUTE PAYMENT (Manual Trigger)
  Frontend Trigger: "Execute" button on dashboard row
  Backend Call:
    POST /api/execute-payment
    Payload: { planId, userAddress }
  Response:
    { transactionHash, status: 'success' | 'failed' }
  UI Flow:
    1. User clicks "Execute" button
    2. Show LoadingSpinner over button
    3. Call backend
    4. If success: Show small success toast ("Payment executed" + hash) + reload table
    5. If failed: Show error toast with specific error message
  Note: For demo, this is manual. In production, backend cron handles it.

FUNCTION 4: PAUSE PLAN
  Frontend Trigger: "Pause" button (or toggle when status = paused)
  Backend Call:
    POST /api/pause-plan
    Payload: { planId, userAddress }
  Response:
    { status: 'paused', nextPayment: null }
  UI Flow:
    1. User clicks "Pause" button
    2. Show LoadingSpinner
    3. Call backend
    4. On success: Reload plans, status badge changes to yellow "Paused", button label flips to "Resume"
    5. On error: Show error toast

FUNCTION 5: RESUME PLAN
  Frontend Trigger: "Resume" button (when status = paused)
  Backend Call:
    POST /api/resume-plan
    Payload: { planId, userAddress }
  Response:
    { status: 'active', nextPayment: ISO datetime }
  UI Flow: Same as pause, but status reverts to green "Active"

FUNCTION 6: CANCEL PLAN
  Frontend Trigger: "Cancel" button
  Confirmation Step:
    Show ConfirmationModal asking "Are you sure? This cannot be undone."
  Backend Call (if confirmed):
    POST /api/cancel-plan
    Payload: { planId, userAddress }
  Response:
    { status: 'cancelled' }
  UI Flow:
    1. User clicks "Cancel" button
    2. Show ConfirmationModal
    3. If user confirms:
       a. Show LoadingSpinner
       b. Call backend
       c. On success: Row disappears from table OR status becomes gray "Cancelled"
       d. Reload stats
    4. If user cancels modal: Modal closes, nothing happens

FUNCTION 7: LOGOUT
  Frontend Trigger: "Sign Out" button in navbar
  Action:
    Clear user state (localStorage, context, etc.)
    Redirect to /
  No backend call needed (demo auth)

SECTION 6: ERROR HANDLING STRATEGY (Trust Building)
================================================================================

ALL API ERRORS SHOULD BE USER-FRIENDLY:

Don't show: "Error 500 Internal Server Error"
Show: "Transaction failed. Check your connection and try again. If it persists, contact support."

ERROR SCENARIOS:
1. Wallet not connected
   Message: "Please connect your wallet to continue."
   Action: Show ConnectWalletButton

2. Insufficient balance
   Message: "You don't have enough USDC to create this plan. Minimum required: [amount]"
   Action: Show amount needed

3. Invalid recipient address
   Message: "Recipient address is not valid. Must be a Stellar address starting with 'G'."
   Action: Focus recipient input field

4. Transaction timeout
   Message: "Transaction took too long. It may still have processed. Check your transaction hash in your wallet."
   Action: Show "Try Again" button

5. Network error
   Message: "Unable to connect to the network. Please check your internet connection."
   Action: Show "Retry" button

SNIPPET 1: Basic Form Pattern
```jsx
const [formData, setFormData] = useState({ recipient: '', amount: '' });
const [errors, setErrors] = useState({});

const handleChange = (e) => {
  setFormData({ ...formData, [e.target.name]: e.target.value });
};

const validateForm = () => {
  const newErrors = {};
  if (!formData.recipient) newErrors.recipient = 'Recipient is required';
  if (formData.amount <= 0) newErrors.amount = 'Amount must be greater than 0';
  return newErrors;
};

const handleSubmit = async (e) => {
  e.preventDefault();
  const newErrors = validateForm();
  if (Object.keys(newErrors).length > 0) {
    setErrors(newErrors);
    return;
  }
  // Call API
};

return (
  <form onSubmit={handleSubmit}>
    <FormInput 
      label="Recipient" 
      name="recipient" 
      value={formData.recipient}
      onChange={handleChange}
      error={errors.recipient}
    />
    <FormInput 
      label="Amount" 
      name="amount" 
      type="number"
      value={formData.amount}
      onChange={handleChange}
      error={errors.amount}
    />
    <button type="submit">Submit</button>
  </form>
);
```

SNIPPET 2: API Call with Loading & Error
```jsx
const [loading, setLoading] = useState(false);
const [error, setError] = useState(null);

const fetchPlans = async () => {
  setLoading(true);
  setError(null);
  try {
    const response = await fetch(`/api/plans?userAddress=${walletAddress}`);
    if (!response.ok) throw new Error('Failed to fetch plans');
    const data = await response.json();
    setPlans(data);
  } catch (err) {
    setError(err.message);
  } finally {
    setLoading(false);
  }
};

useEffect(() => {
  if (walletAddress) fetchPlans();
}, [walletAddress]);

return (
  <div>
    {loading && <LoadingSpinner />}
    {error && <div className="text-red-600">{error}</div>}
    {!loading && plans.length > 0 && <PlansTable plans={plans} />}
    {!loading && plans.length === 0 && <p>No plans found</p>}
  </div>
);
```

SNIPPET 3: Toast Notification (Simple)
```jsx
const [toast, setToast] = useState(null);

const showToast = (message, type = 'success') => {
  setToast({ message, type });
  setTimeout(() => setToast(null), 3000); // Auto-dismiss after 3s
};

return (
  <div>
    {toast && (
      <div className={`fixed bottom-4 right-4 p-4 rounded ${
        toast.type === 'success' ? 'bg-green-500' : 'bg-red-500'
      } text-white`}>
        {toast.message}
      </div>
    )}
  </div>
);
```
